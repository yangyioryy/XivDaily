# Plan: XivDaily 全栈开发计划

🎯 任务概述

本项目要交付一款面向科研人员和 AI 开发者的原生 Android 应用，核心是围绕 arXiv 论文做“可定制时间窗”的快筛、AI 导读、收藏与 Zotero 同步。客户端按 `Kotlin + Jetpack Compose + MVVM + Room/DataStore` 实现，后端按 Python 技术栈独立提供论文聚合、AI 摘要/翻译、Zotero 转发与管理接口。前端视觉与交互实现默认以项目根目录下的 `设计图.png` 为主参考，优先贴近其信息层级、卡片排布、筛选区结构、底部导航和设置页分组样式。

当前仓库还处于初始化状态，因此计划需要覆盖仓库骨架、接口协议、数据模型、客户端三大页面、后端能力、测试验收到交付发布的完整路径。后端开发默认使用 `xivdaily` conda 环境，MySQL 作为服务端持久化主库，客户端本地继续使用 Room 做离线收藏与缓存。开发阶段的大模型密钥来源固定为项目根目录下的 `openai开发密钥.txt`，但仅允许作为本机开发输入源使用，不能提交进仓库，也不能明文写入代码或静态配置。

📋 执行计划

1. 冻结 MVP 边界与系统模型：把 `需求.txt` 与设计图中的交互拆成最小闭环，明确首版只覆盖首页、收藏库、设置页、AI 导读、摘要翻译、Zotero 同步与 BibTeX 导出，先不扩展登录、多端同步、推荐算法等非核心功能。
2. 建立仓库基础结构：创建 `android/`、`backend/`、`docs/`、`plan/`、`scripts/` 等目录，补充顶层 `README`、环境样例配置、统一接口约定文档，保证客户端与后端可以并行开发。
3. 设计领域模型与接口协议：定义 `Paper`、`PaperQuery`、`TrendSummary`、`FavoritePaper`、`SyncRecord`、`UserPreference`、`TranslationTask` 等核心实体，先出 OpenAPI 草案，再反推 Android DTO 与 Room 表结构。
4. 落地 Python 后端基础设施：建议使用 `FastAPI + SQLAlchemy 2 + Alembic + Pydantic + httpx`，先完成配置管理、日志、异常封装、健康检查、MySQL 连接、基础迁移脚本与 `.env` 读取，所有敏感配置通过环境变量注入；开发期可由本机 `openai开发密钥.txt` 转入运行环境，但避免把数据库口令和 API Key 写进仓库。
5. 实现 arXiv 聚合与检索能力：完成分类筛选、关键词检索、时间窗口查询、结果去重、发布日期归一化、分页与缓存策略；若 arXiv 原始接口无法直接满足“最近 N 天”筛选，需要在后端追加日期过滤和本地缓存层。
6. 实现 AI 能力编排层：封装统一的大模型网关，支持“趋势摘要生成”和“单论文摘要翻译”两类任务，定义提示词模板、超时与重试策略、失败降级文案，并记录调用日志以便后续排障和成本评估。
7. 实现 Zotero 集成模块：完成配置校验、单篇同步、批量导出 BibTeX、同步状态回写、重复同步幂等处理；接口层明确 `已同步/未同步/同步失败` 三态，供收藏库列表与首页按钮直接消费。
8. 搭建 Android 客户端骨架：建立 Compose Navigation、Material 3 主题、MVVM 分层、Repository、Retrofit、Room、DataStore、统一 Result 封装、错误态组件与加载态组件；同时落地深浅色主题和中文文案资源组织，并先抽出与 `设计图.png` 对齐的颜色、圆角、阴影、间距和卡片样式令牌。
9. 逐页实现核心功能：先做首页的分类 Chip、时间窗切换、AI 摘要卡片、论文流与卡片操作；再做收藏库的本地持久化、状态标记、批量选择、导出与删除；最后做设置页的偏好项、默认时间窗、Zotero 配置、大模型配置、主题与个人信息表单。页面实现过程中要优先对齐设计图中的层级、留白、图标节奏和状态标识，再在此基础上补可用性细节。
10. 建立测试与发布闭环：后端补单元测试与接口测试，客户端补 ViewModel/Repository 测试与基础 UI 测试，增加联调脚本、假数据夹具、错误场景回归清单，最后产出 APK 可运行版本与部署说明。

📦 阶段里程碑

- Milestone 1：仓库骨架、接口草案、数据库模型、页面信息架构确定，可开始前后端并行开发。
- Milestone 2：后端完成论文查询、AI 摘要/翻译、Zotero 同步接口，客户端完成首页基础浏览与设置页配置录入。
- Milestone 3：收藏库离线能力、批量导出、同步状态回写与主题切换完成，主流程进入联调。
- Milestone 4：完成回归测试、异常处理、性能优化与交付文档，形成首个可演示版本。

⚠️ 风险与注意事项

- arXiv 原始数据源对排序、分页、时间筛选的支持有限，后端必须预留二次过滤与缓存策略，否则首页刷新体验会不稳定。
- AI 摘要和翻译依赖外部模型服务，首版要优先设计超时、限流、重试与降级文案，避免首页被单个摘要任务阻塞。
- `openai开发密钥.txt` 属于本机敏感文件，只能作为开发期密钥来源，后续 `.gitignore`、部署文档和读取脚本都要避免误提交或误打印明文。
- `设计图.png` 当前应视为前端实现的主基准稿；如果后续真实功能与设计稿存在冲突，需要先以功能闭环为准，再做最小视觉调整，而不是随意偏离整体布局。
- Zotero 远程库接口存在权限、限速和字段兼容问题，同步逻辑必须做幂等设计，并提供失败原因回显。
- 客户端同时存在 Room 本地收藏与服务端 MySQL 数据，两边的数据责任边界要先讲清：首版以“本地收藏为主、服务端查询为辅”，不要过早引入双向同步。
- 当前仓库尚未初始化 Android/后端工程，建议实现阶段按“先搭骨架、再补功能、最后联调”的顺序推进，避免一次性跨太多文件导致难以验证。
